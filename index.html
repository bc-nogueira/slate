    Introduction

    The Kickserv API is organized around REST. Our API has predictable resource-oriented URLs, accepts form-encoded request bodies, returns JSON-encoded responses, and uses standard HTTP response codes, authentication, and verbs.

    The Kickserv API doesn't support bulk updates. You can work on only one object per request.

    Base URL

    Production

    https://app.kickserv.com

    Sandbox

    https://staging.kickserv.com

    Authentication

    To authorize, use this code:

    # With shell, you can just pass the correct header with each request
curl --request GET \
  --url https://app.kickserv.com/:subdomain/api/v4/account.json \
  --header 'Authorization: Authorization: Bearer token'

    Make sure to replace token with your JWT.

    To make requests to our API v4 please use the Bearer Token authentication using JWT as the authentication token.

    Authorization: meowmeowmeow

    You must replace token with your JWT.

    Errors

    Kickserv uses conventional HTTP response codes to indicate the success or failure of an API request:

    Error Code Meaning
    200 Ok -- Everything worked as expected
    400 Bad Request -- Your request is invalid.
    401 Unauthorized -- Your API key is wrong.
    403 Forbidden -- The kitten requested is hidden for administrators only.
    404 Not Found -- The specified kitten could not be found.
    500 Internal Server Error -- We had a problem with our server. Try again later.

    Error response

    Error objects provide additional information about problems encountered while performing an operation.

    Attribute Description
    Status The HTTP status code applicable to this problem
    Title Description
    Detail Description
    Links A links object that MAY contain the following members

    After an errror, the server returns a JSON structured like this:

    {
  "status": 404,
  "title": "Error title",
  "detail": "Error detailed message",
  "links": ""
}

-H "Accept: application/json" </span>

    "data": {
